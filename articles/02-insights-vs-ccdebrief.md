# Claude Code shipped `/insights`. So why am I still running `cc-debrief`?

*Two ways to debrief a Claude Code session. Same goal, very different output.*

---

In April 2026 Claude Code shipped a built-in `/insights` slash command. You type `/insights`, Anthropic runs Haiku over your last 30 days of sessions, and you get an HTML report on how you've been working.

It's a great feature. It's also exactly the kind of thing that should make a small open-source tool with a similar pitch — like the one I built, [`cc-debrief`](https://github.com/MichaelSourbron/cc-debrief) — feel suddenly redundant.

So I ran both on the same week of work and looked at what each one actually told me.

They are not the same tool. They are not even trying to be the same tool. The differences matter, and once you see them you stop asking *which one* and start asking *when*.

---

## The two tools, side by side

| | `/insights` (built-in) | `cc-debrief` (open source) |
|---|---|---|
| **Engine** | Haiku LLM | pure deterministic |
| **Token cost** | charges your subscription | zero |
| **Privacy** | sends sessions through the API | 100% local |
| **Scope** | last 30 days of sessions | one session OR cross-session combined |
| **Strengths** | qualitative — sentiment, frustration, themes | quantitative — cache TTL, subagent split, repeated calls, idle gaps |
| **Weaknesses** | costs tokens; requires API access | no qualitative analysis |

Same starting input — your `~/.claude/projects/*/*.jsonl` transcripts — and a wildly different way of looking at them.

---

## What `/insights` is actually good at

`/insights` reads. It reads your prompts, Claude's replies, and the back-and-forth between them, and it tells you what *kind* of collaborator you are.

On my report it surfaced things I genuinely couldn't have computed from token counts:

- The *shape* of my workflow: brainstorm → spec → plan → execute → review → iterate. I knew this in a vague way; seeing it named in a sentence made it actionable.
- The places I get stuck: 20 *"wrong approach"* moments where I had to interrupt and redirect, 13 *"misunderstood request"* moments, 9 *"excessive changes"* events where Claude over-engineered something I'd asked to keep small.
- A genuinely funny finding: in one session Claude rated the confidentiality of some personal data at C4, ignored the redaction context I'd given it, and *invented an entire Integrity rating scale* I'd never asked for. I had to clear the values. `/insights` flagged this as "you catch fabrications and force corrections."

That kind of qualitative read is something only an LLM can do. It read 2,815 messages and gave me back a *narrative*. That's the value proposition.

It also costs subscription tokens to run, and it ships your sessions through the API to do the analysis. For most people on a paid plan that's a perfectly fine trade. For some people (regulated industries, sensitive client work, anyone who works with data they can't send anywhere) that trade is a non-starter.

---

## What `cc-debrief` is actually good at

`cc-debrief` doesn't read your prompts in any meaningful sense. It counts.

It walks the JSONL files, sums tokens by source, computes cache hit rate, finds idle gaps, ranks repeated reads, walks subagent internals, applies pricing math, and renders the result as a static HTML page. Pure local, pure deterministic, 100 ms per session, no LLM in the loop.

The output is the kind of thing an LLM is genuinely *bad* at producing reliably:

- *"You read `database.ts` 155 times. Roughly 36,600 of those tokens were redundant. Here's the `CLAUDE.md` snippet to fix it."*
- *"76 idle gaps exceeded the 5-minute prompt cache TTL. The longest was 50 hours. Each one paid full price for context that should have been cached."*
- *"39 subagents ran 482 internal turns. Here's the per-subagent token breakdown the main transcript hides from you."*
- *"This session would have cost ~$717 on Sonnet 4.6 instead of the ~$3,580 you actually paid."*

These are arithmetic claims. They have one right answer. You don't want them generated by sampling — you want them computed.

The deterministic-LLM split is the whole point. `/insights` is great at *"you keep getting frustrated around test failures"*. `cc-debrief` is great at *"that frustration cost you ~570,000 redundant input tokens this week."*

---

## When to use which

After a week of running both, this is how I think about them:

**Run `/insights` when you want to understand your *behaviour*:**
- *Am I drifting toward over-engineering?*
- *Where do I keep getting frustrated?*
- *What am I best at delegating to subagents?*
- *Are there themes across the last 30 days I'd miss in any one session?*

**Run `cc-debrief` when you want to understand your *cost*:**
- *Where exactly did the tokens go in this 9-day session?*
- *Which files am I re-reading?*
- *What would this have cost on Sonnet?*
- *Can I share this report with a teammate without sending anything to the API?*

I run `/insights` roughly weekly — it's a vibe check on my workflow as a whole. I run `cc-debrief` after any session that *felt* expensive, because it tells me precisely *which* parts were expensive and what to change in `CLAUDE.md` next time.

They don't compete. They're answering different questions.

---

## A note on privacy

This is the one place the two tools genuinely diverge in *kind*, not just *flavour*.

`/insights` runs Haiku over your sessions inside Anthropic's API. That's fine for almost everyone. But if you're working under an NDA, with PHI/PII, with client code under contract, or just on something you'd rather not send anywhere — that's a hard stop, even though Anthropic's data handling is extremely good.

`cc-debrief` runs entirely in your browser (or your terminal). No LLM calls. No API key. No internet. The web version uses `FileReader` + `showDirectoryPicker` so transcripts never leave the page. The resulting `report.html` is a static file you can share with a teammate without a single byte going to a third party.

If "100% local" is a requirement for your work, the choice makes itself.

---

## How to run them

**`/insights`** (you already have it): in any Claude Code session, type:

```
/insights
```

It'll open the report in your browser when it's done.

**`cc-debrief`** (web, zero install):

> [michaelsourbron.github.io/cc-debrief](https://michaelsourbron.github.io/cc-debrief/)

Click *Choose folder*, point at your `~/.claude` directory. Click any session, or `ALL` for a cross-session combined view.

**`cc-debrief`** (CLI):

```bash
npx cc-debrief ~/.claude/projects/<project>/<session>.jsonl
open report.html
```

---

## The takeaway

When Anthropic shipped `/insights` my first instinct was to wonder if `cc-debrief` was now pointless. It isn't, and the reason is interesting: the two tools are at opposite ends of a useful axis.

`/insights` uses an LLM to do the things only LLMs can do — read context, identify patterns in language, name workflows in human terms.

`cc-debrief` uses pure code to do the things LLMs are bad at — exact arithmetic, deterministic ranking, reproducible reports, no network, no surprises.

Use them together. They're each better than either of them is alone.
